PROJECT PROGRESS DESCRIPTION

Project Abstract
Safe-Stay is a full-stack web application for discovering, creating, and managing property listings, with user authentication and review support. The platform allows users to sign up, log in, create listings with image uploads, edit/delete their own listings, and browse listing details. Registered users can also submit reviews and ratings. The backend is built using Node.js, Express.js, and MongoDB (via Mongoose), while the frontend uses EJS templates and static assets. The application integrates Cloudinary for image hosting and uses session-based authentication with Passport.js and Mongo-backed session storage. Current progress includes core CRUD functionality for listings, authentication flows, route-level protection, and form validation middleware. The deployed app link indicates that the product is already accessible in a hosted environment. Remaining work is focused on robustness, bug fixes, improved testing coverage, and feature completeness.

Updated Project Approach and Architecture
The current architecture follows a modular MVC-style organization. Data models are defined with Mongoose schemas (`listing`, `review`, `user`) and relationships are handled through ObjectId references. Express routes are separated by feature (`listings`, `reviews`, `users`) and include async error handling through a wrapper utility. Input validation is implemented with Joi schemas before persistence. Authentication uses Passport Local Strategy with `passport-local-mongoose`, while sessions are stored in MongoDB via `connect-mongo` for persistence and scalability. File uploads are handled using Multer with Cloudinary storage integration (`multer-storage-cloudinary`). Server-side rendering is implemented with EJS and `ejs-mate` layouts. Middleware ensures access control (`isloggedin`, owner checks, redirect memory) and flash messaging communicates user actions. This approach keeps concerns separated, improves maintainability, and supports iterative feature expansion.

Challenges/Roadblocks
The main roadblocks have been around data consistency and route-level reliability. One challenge is ensuring consistent schema naming and relation updates between listings and reviews, especially when extending delete workflows. Another challenge is robust validation and error handling across all forms and routes to avoid runtime crashes from malformed input or missing files. Session and environment setup also required careful handling across local/deployed modes (for secret keys and database URLs). Some code-level issues (for example typos in variable names and incomplete delete-review flow) may cause behavior inconsistencies and require cleanup. To solve these issues, the team is applying strict route testing, targeted refactoring of inconsistent names, and adding manual plus automated validation checks before release. A final pass on UX messages and fail-safe redirects is also planned to improve user-facing reliability.
